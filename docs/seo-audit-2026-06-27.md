# raW — Technical SEO Audit & Action Plan (2026-06-27)

**Site:** https://www.myraw.app
**Stack (verified):** Vite + React 18 SPA, React Router (`BrowserRouter`), deployed on **Vercel** (`framework: "vite"`), Capacitor for the mobile shells. **This is _not_ a Next.js repo** — that changes the tooling recommendations (see §6).
**Scope:** Ethical, white-hat technical SEO only. No bots, fake traffic, fake searches, click farms, scraping abuse, or cloaking.

> **Supersedes / extends** `docs/seo-audit.md` (2026-06-09). Since that pass, these are **already fixed in the current code**: the `theartofraw.me` → `myraw.app` domain mismatch (#1), viewport `maximum-scale` zoom block (#8), sitemap `lastmod`/no-`priority` (#7), and `FAQPage` + `WebSite`/`SearchAction` structured data (#5). The home gate now ships a crawlable `seo-landing` block in `index.html` (#3, partial). The **one big item still open is prerendering/SSR (#2/#4)** — it remains this audit's P0.
>
> **Bottom line:** Strong foundation already in place (robots, sitemap, canonical, per-route meta, JSON-LD, llms.txt, crawlable splash fallback, good headers/caching). The dominant limitation was structural: a **client-rendered SPA** serving the homepage's `<head>` for every route.

> **Update 2026-06-27 (PR #707):** The **P0 is now implemented** — a dependency-free post-build step (`scripts/prerender-seo.mts` + `src/components/seo/prerender.ts`) bakes route-correct `<title>`/description/canonical/OG/JSON-LD into static HTML for all 7 indexable routes (`dist/<route>/index.html`), driven by the same `seo-config.ts` that `RouteSeo` uses at runtime. Non-JS crawlers and social scrapers now get correct per-route metadata. **Remaining limitation:** only the `<head>` is prerendered — subpage *body* content is still SPA-rendered (the homepage-only `seo-landing` block is stripped from subpages to avoid a duplicate `<h1>`). Full body prerendering remains a later enhancement.

### The one structural issue behind most 🔴/🟡 above
`index.html` is a single static shell. Every route is rewritten to it (`vercel.json` `rewrites`), and `RouteSeo.tsx` patches `<title>`/meta/canonical/JSON-LD in a `useEffect` **after JS runs**. Consequences:

- **Googlebot** renders JS, so it mostly sees the corrected tags — OK but not ideal (render budget, delayed canonical).
- **Bing** (limited JS), **social scrapers** (FB/X/LinkedIn/Slack/WhatsApp — no JS), and **most LLM crawlers** see the **homepage** title, description, canonical (`/`), and OG for *every* URL.
- Net effect: deep-page share cards show homepage content, and deep pages risk being collapsed into the homepage as duplicates.

**→ P0 fix: prerender the 7 marketing routes to static HTML at build time (§8).**

### Recommended P0 code — prerender setup (`@prerenderer`, zero app refactor)

> Builds the SPA, then loads each route in headless Chromium and writes the fully-rendered HTML (with `RouteSeo`'s corrected `<head>`) to `dist/<route>/index.html`. No change to the React entry beyond firing one event.

```bash
npm i -D @prerenderer/rollup-plugin @prerenderer/renderer-puppeteer
```

```ts
// vite.config.ts — add to plugins
import prerender from '@prerenderer/rollup-plugin'

plugins: [
  // ...existing...
  prerender({
    routes: [
      '/', '/communities-explained', '/polls-explained',
      '/faq', '/security', '/privacy', '/terms',
    ],
    renderer: '@prerenderer/renderer-puppeteer',
    rendererOptions: { renderAfterDocumentEvent: 'prerender-ready', headless: true },
  }),
]
```

```ts
// src/main.tsx — signal readiness once React + RouteSeo have patched <head>
requestAnimationFrame(() => document.dispatchEvent(new Event('prerender-ready')))
```

The snapshot captures whatever `RouteSeo` set, so canonical, title, description, OG, and JSON-LD are all baked per page. Verify post-deploy:

```bash
curl -s https://www.myraw.app/polls-explained | grep -E 'canonical|og:title|<title>'
```

> Alternative (cleaner long-term): migrate to **`vite-react-ssg`** (renders routes to static HTML + auto-sitemap) — requires converting `App.tsx`'s `<Routes>` to its route config. Choose this if many pages are coming.
